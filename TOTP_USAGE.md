# TOTP动态验证码使用说明

## 📋 功能介绍

TOTP（Time-based One-Time Password，基于时间的一次性密码）是一种广泛使用的双因素认证技术。本工具现在采用类似Google Authenticator的TOTP算法，每30秒自动生成一个新的6位数字验证码。

## 🔐 核心特性

### 1. 自动刷新
- ⏰ **每30秒刷新一次**：验证码会自动更新
- 🔄 **无缝切换**：新旧验证码之间有重叠期
- 📊 **实时倒计时**：显示下次刷新的剩余时间

### 2. 时间窗口容错
- ✅ **当前窗口**：当前30秒内的验证码有效
- ✅ **前一个窗口**：前30秒的验证码仍然有效
- ✅ **后一个窗口**：后30秒的验证码提前有效
- 🎯 **总共90秒有效期**：提供充足的操作时间

### 3. 安全性
- 🔒 **HMAC-SHA1算法**：行业标准加密算法
- 🔑 **唯一密钥**：每个应用有独立的密钥
- ⚡ **不可预测**：基于时间和密钥生成，无法推测
- 🛡️ **防重放攻击**：验证码过期后失效

## 💡 工作原理

### TOTP算法流程

```
1. 获取当前时间戳
   ↓
2. 计算时间计数器 = 当前时间 / 30秒
   ↓
3. 使用HMAC-SHA1(密钥, 时间计数器)
   ↓
4. 动态截断得到6位数字
   ↓
5. 生成验证码
```

### 时间窗口机制

```
时间轴：  ... |---T-1---|---T0---|---T+1---| ...
          (前窗口)  (当前)  (后窗口)

验证时检查3个窗口：
- T-1: 前30秒的验证码
- T0:  当前30秒的验证码（主要）
- T+1: 后30秒的验证码

只要匹配其中任何一个，验证就通过
```

### 为什么需要时间窗口？

1. **网络延迟**：客户端和服务器时间可能不同步
2. **用户操作**：给用户充足的时间输入
3. **时钟漂移**：不同设备的时钟可能有误差
4. **用户体验**：避免因为几秒钟的误差导致验证失败

## 📊 验证码生命周期

### 单个验证码的有效期

```
时间 0s:    验证码A开始有效
时间 30s:   验证码B开始有效，A仍然有效（前一窗口）
时间 60s:   验证码C开始有效，B仍然有效，A失效
时间 90s:   验证码D开始有效，C仍然有效，B失效
```

### 实际例子

假设现在是 12:00:00：

| 时间段 | 有效验证码 | 说明 |
|--------|-----------|------|
| 11:59:30 - 12:00:00 | ABC123 | 前一个窗口的验证码 |
| 12:00:00 - 12:00:30 | DEF456 | 当前窗口的验证码（主要） |
| 12:00:30 - 12:01:00 | GHI789 | 下一个窗口的验证码（提前有效） |

在 12:00:15 时，ABC123、DEF456、GHI789 都可以成功验证！

## ⚙️ 技术配置

### 可调整参数

在 `wecom/utils/daily_password.py` 中可以修改：

```python
# TOTP配置
TOTP_TIME_STEP = 30  # 时间步长（秒），默认30秒
TOTP_DIGITS = 6      # 密码位数，默认6位
TOTP_WINDOW = 1      # 时间窗口数量，默认前后各1个

# 密钥（建议在生产环境中保密存储）
SECRET_KEY = b'WeComBatchSender2024SecretKey'
```

### 参数说明

| 参数 | 默认值 | 说明 | 建议范围 |
|------|--------|------|----------|
| TOTP_TIME_STEP | 30秒 | 刷新间隔 | 30-60秒 |
| TOTP_DIGITS | 6位 | 密码长度 | 6-8位 |
| TOTP_WINDOW | 1 | 窗口数量 | 0-2 |

### 自定义密钥

为了更高的安全性，可以修改密钥：

```python
import os

# 生成随机密钥（只需执行一次）
secret = os.urandom(32)
print(f"SECRET_KEY = {secret}")

# 将生成的密钥复制到配置中
SECRET_KEY = b'your_generated_secret_here'
```

**注意：** 修改密钥后，所有之前生成的验证码都会失效！

## 🔍 常见问题

### Q1: 为什么验证码会变化？

**A:** 这是TOTP的设计特性。每30秒基于当前时间重新计算，确保安全性。

### Q2: 如果我在验证码即将刷新时输入怎么办？

**A:** 不用担心！系统会检查前后各30秒的验证码，即使在你输入的过程中刷新了，仍然可以验证通过。

### Q3: 多台设备上的验证码一样吗？

**A:** 是的！只要：
- 使用相同的密钥
- 系统时间准确
- 在同一时间窗口内

所有设备生成的验证码都相同。

### Q4: 系统时间不准确会影响验证吗？

**A:** 有一定影响，但时间窗口机制可以容忍最多±30秒的误差。建议保持系统时间准确。

### Q5: 可以和Google Authenticator一起用吗？

**A:** 理论上可以，但需要：
1. 使用相同的密钥
2. 配置相同的参数（30秒，6位）
3. 将密钥导入Google Authenticator

目前本工具使用内置密钥，暂不支持外部导入。

### Q6: 验证失败了怎么办？

**A:** 可能的原因：
1. 输入的验证码已过期（超过90秒）
2. 系统时间不准确
3. 输入错误

解决方法：
- 重新获取最新验证码
- 检查系统时间
- 仔细输入6位数字

### Q7: 这个比每日密码好在哪里？

**A:** TOTP的优势：
- ✅ 更高的安全性（频繁变化）
- ✅ 更短的有效期（减少泄露风险）
- ✅ 行业标准（广泛使用）
- ✅ 更好的用户体验（有倒计时提示）

## 🎯 最佳实践

### 使用建议

1. **及时输入**
   - 看到验证码后尽快输入
   - 避免在即将刷新时输入
   - 最好在刷新后的前20秒内完成

2. **保持时间准确**
   - 启用Windows自动时间同步
   - 定期检查系统时间
   - 使用NTP服务器同步

3. **妥善保管**
   - 不要截图分享验证码
   - 不要通过不安全渠道传输
   - 验证码只在90秒内有效

4. **熟悉界面**
   - 注意观察倒计时
   - 剩余5秒时会显示警告
   - 利用一键复制功能

### 安全建议

⚠️ **重要提醒：**

1. **密钥保密**
   - SECRET_KEY不应该公开
   - 不要在代码仓库中提交密钥
   - 生产环境应该使用环境变量

2. **时间同步**
   - 确保服务器和客户端时间一致
   - 使用可靠的时间源
   - 定期校准系统时钟

3. **监控异常**
   - 记录验证失败的尝试
   - 检测暴力破解行为
   - 实施速率限制

## 🔧 故障排查

### 问题1：验证码总是验证失败

**检查项：**
1. 系统时间是否准确
2. 是否正确输入6位数字
3. 是否在90秒有效期内

**解决方法：**
```bash
# Windows时间同步
w32tm /resync

# 重新获取验证码
python password_generator.py
```

### 问题2：倒计时不准确

**检查项：**
1. 系统负载是否过高
2. Python进程是否正常
3. 是否有其他程序干扰

**解决方法：**
- 重启密码生成器
- 检查系统资源使用情况
- 关闭不必要的程序

### 问题3：GUI界面不更新

**检查项：**
1. tkinter是否正常
2. 事件循环是否运行
3. 是否有异常错误

**解决方法：**
```bash
# 测试tkinter
python -c "import tkinter; print('tkinter OK')"

# 重新启动GUI
python password_generator_gui.py
```
